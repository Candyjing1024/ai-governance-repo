================================================================================
  MODEL DEPLOYMENT REQUEST WORKFLOW — REST API GUIDE (Foundry V2)
================================================================================

  Purpose:  Document every REST API call needed to build a Model Deployment
            Request workflow (submit → review → approve/reject → deploy).
  Format:   Language-agnostic curl commands. Copy into any HTTP client.
  Storage:  Azure Cosmos DB (NoSQL)
  Deploy:   Azure AI Foundry V2 (AIServices kind) + APIM AI Gateway (optional)

  API Versions (latest stable as of 2025-04-24):
    - Foundry / CognitiveServices ARM: 2024-10-01 (latest GA)
    - Cosmos DB ARM:                   2024-11-15
    - Cosmos DB data-plane:            2018-12-31 (x-ms-version header)
    - APIM ARM:                        2024-06-01-preview (AI Gateway features)
    - Microsoft Graph:                 v1.0

  Author:   AI Governance Team
  Date:     2026-04-22 (Updated 2025-04-24 — migrated to Foundry V2 + latest API versions)

================================================================================
  0. ARCHITECTURE OVERVIEW
================================================================================

  The flow has 4 actors and 3 systems:

  ┌────────────┐      ┌────────────────┐      ┌────────────────┐
  │   User /   │      │  Backend API   │      │   Azure        │
  │   Chat UI  │─────>│  (your app)    │─────>│   Cosmos DB    │
  └────────────┘      └───────┬────────┘      └────────────────┘
                              │
                   On approve │
                              ▼
                    ┌─────────────────┐      ┌────────────────┐
                    │  Azure AI       │      │  Azure APIM    │
                    │  Foundry (ARM)  │      │  (optional)    │
                    └─────────────────┘      └────────────────┘

  Flow:

  1. User says "add models" in chat UI
     → UI renders a form (Model name, Deployment name, Project, etc.)

  2. User fills form, clicks "Submit request"
     → Backend POSTs document to Cosmos DB
     → Status = "requested_pending_approval"
     → Returns request ID to user

  3. Admin says "review models" in chat UI
     → Backend queries Cosmos for all requests
     → UI renders table with Approve / Reject buttons

  4. Admin clicks "Approve"
     → Backend PATCHes Cosmos status to "approved"
     → Backend triggers model deployment to Foundry via ARM REST API
     → (Optional) Backend adds APIM operation for the new deployment

  5. Admin clicks "Reject"
     → Backend PATCHes Cosmos status to "rejected"


================================================================================
  1. PREREQUISITES — GET AUTH TOKENS
================================================================================

  You need TWO tokens:
    A) ARM token — for Cosmos DB management, Foundry deployment, APIM
    B) Cosmos DB data-plane token — for document CRUD

  The Cosmos DB REST API uses either:
    - A master key (primary key from Azure portal), OR
    - Entra ID (AAD) tokens with RBAC

  This guide shows BOTH approaches.

--------------------------------------------------------------------------
  1A. Get ARM Token (Service Principal)
--------------------------------------------------------------------------

  Replace these variables in ALL commands below:

  TENANT_ID="<your-entra-tenant-id>"
  CLIENT_ID="<your-sp-client-id>"
  CLIENT_SECRET="<your-sp-client-secret>"
  SUBSCRIPTION_ID="<your-azure-subscription-id>"
  RESOURCE_GROUP="<your-resource-group>"

  # Get ARM token
  curl -s -X POST \
    "https://login.microsoftonline.com/${TENANT_ID}/oauth2/v2.0/token" \
    -H "Content-Type: application/x-www-form-urlencoded" \
    -d "grant_type=client_credentials&client_id=${CLIENT_ID}&client_secret=${CLIENT_SECRET}&scope=https://management.azure.com/.default"

  # Response:
  # { "access_token": "<ARM_TOKEN>", "expires_in": 3599, ... }

  ARM_TOKEN="<paste-access-token-here>"

--------------------------------------------------------------------------
  1B. Get Cosmos DB Account Info
--------------------------------------------------------------------------

  Once you create the Cosmos account (Section 2), get the primary key:

  COSMOS_ACCOUNT="<your-cosmos-account-name>"
  COSMOS_ENDPOINT="https://${COSMOS_ACCOUNT}.documents.azure.com"

  # Get Cosmos keys via ARM
  curl -s -X POST \
    "https://management.azure.com/subscriptions/${SUBSCRIPTION_ID}/resourceGroups/${RESOURCE_GROUP}/providers/Microsoft.DocumentDB/databaseAccounts/${COSMOS_ACCOUNT}/listKeys?api-version=2024-11-15" \
    -H "Authorization: Bearer ${ARM_TOKEN}" \
    -H "Content-Type: application/json"

  # Response:
  # { "primaryMasterKey": "<COSMOS_KEY>", ... }

  COSMOS_KEY="<paste-primary-key-here>"

--------------------------------------------------------------------------
  1C. Cosmos DB Authorization Header
--------------------------------------------------------------------------

  The Cosmos DB REST API requires a special Authorization header:

    Authorization: type=master&ver=1.0&sig=<base64-hmac-sha256>

  The signature is computed from:
    - HTTP verb (GET, POST, PUT, etc.)
    - Resource type (dbs, colls, docs)
    - Resource link (dbs/ModelRequests, dbs/ModelRequests/colls/Requests, etc.)
    - UTC date (x-ms-date header)
    - The master key

  GENERATING THE AUTH HEADER:

  Since this is complex to do in pure bash, here's a helper function.
  Paste this in your terminal before running the curl commands:

  ──────────────────────────────────────────────────────────────────
  generate_cosmos_auth() {
    # Usage: generate_cosmos_auth <verb> <resourceType> <resourceLink> <date> <key>
    local VERB=$(echo -n "$1" | tr '[:upper:]' '[:lower:]')
    local RESOURCE_TYPE=$(echo -n "$2" | tr '[:upper:]' '[:lower:]')
    local RESOURCE_LINK="$3"
    local DATE="$4"
    local KEY="$5"

    local PAYLOAD="${VERB}\n${RESOURCE_TYPE}\n${RESOURCE_LINK}\n${DATE}\n\n"
    local HEX_KEY=$(echo -n "$KEY" | base64 -d | xxd -p -c 256)
    local SIG=$(printf "${PAYLOAD}" | openssl dgst -sha256 -mac HMAC \
      -macopt "hexkey:${HEX_KEY}" -binary | base64)
    echo -n "type=master&ver=1.0&sig=${SIG}" | python3 -c "import sys,urllib.parse; print(urllib.parse.quote(sys.stdin.read(), safe=''))"
  }
  ──────────────────────────────────────────────────────────────────

  NOTE: Every curl command below shows <AUTH_HEADER> as a placeholder.
  Before each call, run:

    DATE=$(TZ=GMT date '+%a, %d %b %Y %H:%M:%S GMT')
    AUTH=$(generate_cosmos_auth "<VERB>" "<resourceType>" "<resourceLink>" "$DATE" "$COSMOS_KEY")

  Then use: -H "Authorization: ${AUTH}" -H "x-ms-date: ${DATE}"

  2. COSMOS DB SETUP (DATABASE + CONTAINER)

--------------------------------------------------------------------------
  2A. Create Cosmos DB Account (ARM REST API)
--------------------------------------------------------------------------

  Endpoint: PUT https://management.azure.com/subscriptions/{sub}/resourceGroups/{rg}
                /providers/Microsoft.DocumentDB/databaseAccounts/{accountName}
                ?api-version=2024-11-15

  curl -s -X PUT \
    "https://management.azure.com/subscriptions/${SUBSCRIPTION_ID}/resourceGroups/${RESOURCE_GROUP}/providers/Microsoft.DocumentDB/databaseAccounts/${COSMOS_ACCOUNT}?api-version=2024-11-15" \
    -H "Authorization: Bearer ${ARM_TOKEN}" \
    -H "Content-Type: application/json" \
    -d '{
      "location": "eastus",
      "kind": "GlobalDocumentDB",
      "properties": {
        "databaseAccountOfferType": "Standard",
        "locations": [
          {
            "locationName": "eastus",
            "failoverPriority": 0
          }
        ],
        "consistencyPolicy": {
          "defaultConsistencyLevel": "Session"
        },
        "capabilities": [
          { "name": "EnableServerless" }
        ]
      }
    }'

  # Response: 200/201 — account provisioning starts
  # Poll until provisioningState = "Succeeded" (~2-5 minutes)

  # Check status:
  curl -s \
    "https://management.azure.com/subscriptions/${SUBSCRIPTION_ID}/resourceGroups/${RESOURCE_GROUP}/providers/Microsoft.DocumentDB/databaseAccounts/${COSMOS_ACCOUNT}?api-version=2024-11-15" \
    -H "Authorization: Bearer ${ARM_TOKEN}" \
    | python3 -c "import sys,json; print(json.load(sys.stdin)['properties']['provisioningState'])"

--------------------------------------------------------------------------
  2B. Create Database: "ModelRequests"
--------------------------------------------------------------------------

  Endpoint: POST https://{account}.documents.azure.com/dbs
  Resource type: dbs
  Resource link: (empty for creating a db)

  DATE=$(TZ=GMT date '+%a, %d %b %Y %H:%M:%S GMT')
  AUTH=$(generate_cosmos_auth "post" "dbs" "" "$DATE" "$COSMOS_KEY")

  curl -s -X POST \
    "${COSMOS_ENDPOINT}/dbs" \
    -H "Authorization: ${AUTH}" \
    -H "x-ms-date: ${DATE}" \
    -H "x-ms-version: 2018-12-31" \
    -H "Content-Type: application/json" \
    -d '{
      "id": "ModelRequests"
    }'

  # Response: 201 Created
  # {
  #   "id": "ModelRequests",
  #   "_rid": "...",
  #   "_self": "dbs/ModelRequests/",
  #   ...
  # }

--------------------------------------------------------------------------
  4A. List ALL requests (any status)
--------------------------------------------------------------------------

  Endpoint: POST https://{account}.documents.azure.com
                 /dbs/ModelRequests/colls/Requests/docs
  Resource type: docs
  Resource link: dbs/ModelRequests/colls/Requests
  IMPORTANT: Set x-ms-documentdb-isquery: True and enable cross-partition

  DATE=$(TZ=GMT date '+%a, %d %b %Y %H:%M:%S GMT')
  AUTH=$(generate_cosmos_auth "post" "docs" "dbs/ModelRequests/colls/Requests" "$DATE" "$COSMOS_KEY")

  curl -s -X POST \
    "${COSMOS_ENDPOINT}/dbs/ModelRequests/colls/Requests/docs" \
    -H "Authorization: ${AUTH}" \
    -H "x-ms-date: ${DATE}" \
    -H "x-ms-version: 2018-12-31" \
    -H "Content-Type: application/query+json" \
    -H "x-ms-documentdb-isquery: True" \
    -H "x-ms-documentdb-query-enablecrosspartition: True" \
    -d '{
      "query": "SELECT c.id, c.model_name, c.deployment_name, c.project_name, c.foundry_resource, c.resource_group, c.tpm, c.region, c.status, c.requested_by, c.requested_at, c.reviewed_by, c.reviewed_at FROM c ORDER BY c.requested_at DESC"
    }'

================================================================================
  5. APPROVE A REQUEST (PATCH → Cosmos + trigger deploy)
================================================================================

  When the admin clicks "Approve", your backend does TWO things:
    A) Update status in Cosmos DB to "approved"
    B) Trigger model deployment to Foundry (Section 7)

--------------------------------------------------------------------------
  5A. Update document status to "approved"
--------------------------------------------------------------------------

  Cosmos DB supports partial updates via PATCH (JSON Patch operations).

  Endpoint: PATCH https://{account}.documents.azure.com
                  /dbs/ModelRequests/colls/Requests/docs/{id}
  Resource type: docs
  Resource link: dbs/ModelRequests/colls/Requests/docs/{id}

  REQUEST_ID="98ded938-f3f2-4e40-91b8-b85ec93aaa16"
  PROJECT_NAME="my-chainlit-app"
  REVIEW_TIMESTAMP=$(date -u +"%Y-%m-%dT%H:%M:%SZ")

  DATE=$(TZ=GMT date '+%a, %d %b %Y %H:%M:%S GMT')
  AUTH=$(generate_cosmos_auth "patch" "docs" "dbs/ModelRequests/colls/Requests/docs/${REQUEST_ID}" "$DATE" "$COSMOS_KEY")

  curl -s -X PATCH \
    "${COSMOS_ENDPOINT}/dbs/ModelRequests/colls/Requests/docs/${REQUEST_ID}" \
    -H "Authorization: ${AUTH}" \
    -H "x-ms-date: ${DATE}" \
    -H "x-ms-version: 2018-12-31" \
    -H "Content-Type: application/json" \
    -H "x-ms-documentdb-partitionkey: [\"${PROJECT_NAME}\"]" \
    -d "{
      \"operations\": [
        { \"op\": \"set\", \"path\": \"/status\", \"value\": \"approved\" },
        { \"op\": \"set\", \"path\": \"/reviewed_by\", \"value\": \"admin@company.com\" },
        { \"op\": \"set\", \"path\": \"/reviewed_at\", \"value\": \"${REVIEW_TIMESTAMP}\" }
      ]
    }"

  # Response: 200 OK — document updated

================================================================================
  7. ON APPROVAL: DEPLOY MODEL TO FOUNDRY V2 (PUT → ARM)
================================================================================

  This is the ARM REST API to deploy a model on a Foundry V2 (AIServices kind)
  account. Uses api-version 2024-10-01 (latest GA for Microsoft.CognitiveServices).
  This is the same API used in our foundry-apim-e2e/02-deploy-model.py.

  Endpoint: PUT https://management.azure.com
                /subscriptions/{sub}/resourceGroups/{rg}
                /providers/Microsoft.CognitiveServices/accounts/{account}
                /deployments/{deploymentName}
                ?api-version=2024-10-01

  Variables from the approved request:

  FOUNDRY_ACCOUNT="my-chubbdemoai-resource"   # from request.foundry_resource
  DEPLOYMENT_NAME="gpt-4o-mini-prod"          # from request.deployment_name
  MODEL_NAME="gpt-4o-mini"                    # from request.model_name
  MODEL_VERSION="2024-07-18"                  # look up latest version for model
  SKU="GlobalStandard"                        # standard SKU
  TPM=100000                                  # from request.tpm
  RG="my-foundry-rg"                          # from request.resource_group

  curl -s -X PUT \
    "https://management.azure.com/subscriptions/${SUBSCRIPTION_ID}/resourceGroups/${RG}/providers/Microsoft.CognitiveServices/accounts/${FOUNDRY_ACCOUNT}/deployments/${DEPLOYMENT_NAME}?api-version=2024-10-01" \
    -H "Authorization: Bearer ${ARM_TOKEN}" \
    -H "Content-Type: application/json" \
    -d "{
      \"sku\": {
        \"name\": \"${SKU}\",
        \"capacity\": ${TPM}
      },
      \"properties\": {
        \"model\": {
          \"format\": \"OpenAI\",
          \"name\": \"${MODEL_NAME}\",
          \"version\": \"${MODEL_VERSION}\"
        }
      }
    }"

  # Response: 201 Created (or 200 if updating existing)
  #
  # Poll for completion:
  # GET same URL → check properties.provisioningState = "Succeeded"

  POLL UNTIL DEPLOYED:

  while true; do
    STATE=$(curl -s \
      "https://management.azure.com/subscriptions/${SUBSCRIPTION_ID}/resourceGroups/${RG}/providers/Microsoft.CognitiveServices/accounts/${FOUNDRY_ACCOUNT}/deployments/${DEPLOYMENT_NAME}?api-version=2024-10-01" \
      -H "Authorization: Bearer ${ARM_TOKEN}" \
      | python3 -c "import sys,json; print(json.load(sys.stdin).get('properties',{}).get('provisioningState','?'))")
    echo "Deployment state: ${STATE}"
    if [ "$STATE" = "Succeeded" ]; then break; fi
    sleep 15
  done

  MODEL VERSION LOOKUP:

  If you need to find the latest version for a given model name:

  curl -s \
    "https://management.azure.com/subscriptions/${SUBSCRIPTION_ID}/providers/Microsoft.CognitiveServices/locations/eastus/models?api-version=2024-10-01" \
    -H "Authorization: Bearer ${ARM_TOKEN}" \
    | python3 -c "
import sys, json
data = json.load(sys.stdin)
for m in data.get('value', []):
    if m['model']['name'] == 'gpt-4o-mini':
        skus = m['model'].get('skus', [])

================================================================================
  NOTES FOR DEVELOPERS
================================================================================

  1. COSMOS AUTH HEADER
     The Cosmos DB data-plane REST API requires HMAC-SHA256 auth headers.
     Most SDKs (Python azure-cosmos, .NET, Java, Node.js) handle this
     automatically. The generate_cosmos_auth bash function above is only
     needed for raw curl calls.

     SDK equivalents:
       Python:  from azure.cosmos import CosmosClient
       .NET:    using Microsoft.Azure.Cosmos;
       Node.js: const { CosmosClient } = require("@azure/cosmos");
       Java:    com.azure.cosmos.CosmosClient

  2. PARTITION KEY
     The x-ms-documentdb-partitionkey header must match the document's
     partition key value. Format: ["value"] (JSON array with string).
     For cross-partition queries, use x-ms-documentdb-query-enablecrosspartition.

  3. IDEMPOTENCY
     Model deployments (PUT) are idempotent — re-running with the same
     deployment name updates the existing deployment rather than creating
     a duplicate.

  4. ERROR HANDLING
     - Cosmos 409: Document with that ID already exists
     - Cosmos 404: Document or container not found
     - ARM 409: Deployment already exists (check current state)
     - ARM 403: Missing RBAC role (need Cognitive Services Contributor)

  5. SECURITY
     - Never expose COSMOS_KEY in client-side code or chat responses
     - Use Entra ID RBAC (Cosmos DB Built-in Data Contributor role) for
       production instead of master keys
     - Validate user roles before allowing approve/reject operations
     - The ARM_TOKEN (SP credentials) should only be used server-side

  6. ALTERNATIVE: COSMOS DB SDK (recommended for production)
     Instead of raw REST, use the SDK which handles auth, retries, and
     pagination automatically:

     Python example:
       from azure.cosmos import CosmosClient, PartitionKey
       client = CosmosClient(endpoint, credential=key)
       db = client.get_database_client("ModelRequests")
       container = db.get_container_client("Requests")
       container.create_item(body={...})
       items = container.query_items("SELECT * FROM c WHERE c.status=@s",
                                     parameters=[{"name":"@s","value":"requested_pending_approval"}],
                                     enable_cross_partition_query=True)
